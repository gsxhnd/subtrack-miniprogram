# 从 Taro 切换到原生微信小程序开发

## 基本信息
- 日期：2026-03-19
- 相关技术：Taro / 微信小程序 / weapp-vite / Vue 3
- 严重程度：中
- 解决状态：已决策

## 问题描述
项目初期考虑使用 Taro 框架进行跨平台开发，但经过技术调研和实际体验后，决定切换到原生微信小程序开发方案（weapp-vite + Vue 3）。

## 环境信息
- 操作系统：macOS 15.7.4
- 原方案：Taro 4.x + React/Vue
- 新方案：weapp-vite + Vue 3 + TypeScript

## 决策过程

### 1. 使用 Taro 的优势（放弃的原因）
- 跨平台能力：一套代码可以编译到多个平台（微信、支付宝、抖音等）
- 生态丰富：可以使用 React/Vue 的生态系统
- 开发体验：熟悉的 Web 开发模式

### 2. 放弃 Taro 的主要原因
生成的代码无法运行

#### 2.1 编译层复杂性
- Taro 需要在源代码和微信小程序代码之间增加一层编译转换
- 调试时需要理解编译后的代码，增加心智负担
- 遇到问题时难以判断是 Taro 层还是原生层的问题

#### 2.2 性能考量
- 额外的运行时适配层会增加包体积
- 某些场景下性能不如原生实现
- 小程序本身已经有诸多限制，再加一层抽象可能雪上加霜

#### 2.3 项目实际需求
- 当前项目主要目标平台是微信小程序
- 暂时没有强烈的多平台需求
- 原生开发可以更好地利用微信最新的能力和特性

#### 2.4 维护成本
- Taro 版本更新可能带来 Breaking Changes
- 需要持续跟进 Taro 的更新和适配
- 原生小程序 API 更新后，Taro 适配需要时间

### 3. 选择 weapp-vite + Vue 3 的理由

#### 3.1 weapp-vite 的优势
- 基于 Vite，开发体验好，热更新快
- 编译目标直接是原生小程序代码，无额外运行时
- 可以使用 Vue 3 的 Composition API，代码组织更灵活
- TypeScript 支持良好

#### 3.2 Vue 3 的优势
- 响应式系统比 Vue 2 更高效
- Composition API 适合逻辑复用
- 类型支持更好

#### 3.3 与原生小程序的结合
- 可以直接使用微信小程序的 API 和组件
- 无需学习额外的框架特定语法
- 调试时直接看编译后的原生代码，更清晰

## 最终决策

采用 **weapp-vite + Vue 3 + TypeScript** 作为项目技术栈：

1. 保持原生小程序的开发体验
2. 获得 Vue 3 和 TypeScript 的开发效率提升
3. 避免 Taro 带来的额外复杂性和性能开销
4. 为未来可能的多平台需求保留灵活性（weapp-vite 也支持其他平台）

## 经验总结

技术选型应该基于项目实际需求，而不是追求"大而全"。当项目的主要目标平台明确时，原生方案往往比跨平台框架更简洁、可控。weapp-vite 在保持原生开发优势的同时，提供了现代化的开发体验，是一个不错的平衡点。
